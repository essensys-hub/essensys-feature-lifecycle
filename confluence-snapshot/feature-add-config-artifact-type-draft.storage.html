Authentification

Les endpoints d'écriture nécessitent le header X-API-Key :

X-API-Key: <SKILLS_API_KEY>

Les endpoints de lecture sont ouverts (pas d'auth requise).

Conventions

  • Réponses JSON (Content-Type: application/json)
  • Erreurs retournées avec code HTTP + {"error": "...", "message": "..."}
  • Paramètre ?tool= optionnel sur la plupart des endpoints (tracking audit)
  • Paramètre ?context= optionnel (filtre visibilité : admin pour voir les items REJECTED)

Architecture des routes

Les routes suivent le principe Interface Adapters de Clean Architecture : elles ne contiennent aucune logique métier et délèguent systématiquement aux services de la couche Application.

Route (api/routes/*.py) → Service (services/*.py) → Model (database/models.py)

POST /api/skills (auth)

Crée une skill avec sa première version, ou crée automatiquement une nouvelle version si un artefact avec le même nom existe déjà (upsert). Déclenche un scan IA (Bedrock) automatique.

Comportement upsert : si name correspond à un artefact existant (y compris REJECTED), l'API : 1. Récupère le skill existant via get_skill_by_name() (sans filtre de visibilité) 2. Bump la version minor (ex: 1.1.0 → 1.2.0) 3. Crée la nouvelle version avec le contenu fourni 4. Retourne 200 OK avec "upserted": true et "new_version": "1.2.0"

Body :

{
  "name": "my-skill",
  "description": "A useful skill",
  "artifact_type": "skill",
  "version": "1.0.0",
  "content": "# My Skill\n\nContent...",
  "changelog": "Initial release",
  "files": {"reference.md": "# Reference..."},
  "is_official": false,
  "source_repo": ""
}

Validation spéciale pour config : quand artifact_type vaut config, le champ content est parsé comme JSON et validé par un JSON Schema (api/validators/config_schema.json). Les champs requis sont name et version (format X.Y). En cas d'erreur de validation, la réponse est 400 Bad Request avec le détail des violations.

Réponses : - 201 Created : nouveau skill créé - 200 OK : skill existant, nouvelle version ajoutée (upsert) - 400 Bad Request : validation échouée (config JSON invalide, nom invalide, etc.)

GET /api/skills/<skill_id>/versions/<vid>/signature

Retourne les informations d'intégrité et de signature pour une version donnée.

Réponse 200 :

{
  "skill_id": 123,
  "version_id": 456,
  "version": "1.0.0",
  "content_hash_sha256": "abc123...",
  "signature": "MEYCIQCMntDu...",
  "signing_key_id": "dev-key-1",
  "algorithm": "ECDSA_SHA_256",
  "signed_at": "2026-02-24T10:00:00+00:00"
}
  • signature : signature ECDSA P-256 encodée base64 (DER). null si la signature n'est pas activée.
  • signing_key_id : identifiant de la clé utilisée pour signer.
  • algorithm : toujours ECDSA_SHA_256 quand une signature est présente.

Réponse 404 : Version non trouvée.

POST /api/skills/<skill_id>/versions/<vid>/verify-integrity

Vérifie l'intégrité d'un artefact : compare le hash client vs serveur et valide la signature.

Body :

{
  "client_hash": "abc123..."
}

Réponse 200 / 409 :

{
  "skill_id": 123,
  "version_id": 456,
  "server_hash": "abc123...",
  "client_hash": "abc123...",
  "hash_match": true,
  "signature_valid": true,
  "trust_level": "SIGNED_VERIFIED"
}
Trust Level Signification
SIGNED_VERIFIED Hash match + signature valide
HASH_ONLY Hash match, pas de signature
SIGNED_UNVERIFIED Signature présente, pas de hash client
TAMPERED Hash mismatch → HTTP 409
SIGNATURE_INVALID Signature invalide ou clé révoquée
UNKNOWN Données insuffisantes

POST /api/profiles/<profile_id>/versions/<vid>/publish (auth)

Publie une version de profile (draft → published).

La publication échoue avec 400 Bad Request si la version contient des items non installables, par exemple :

  • un item latest sans version stable (CLEAN ou APPROVED_BY_HUMAN)
  • un item pinned dont la version demandée est introuvable

Dans ce cas, le corps JSON contient un message détaillant les artefacts bloquants.
