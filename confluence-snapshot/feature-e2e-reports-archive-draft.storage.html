E2E Testing Suite

Comprehensive end-to-end test suite for the Skills Portal React application (wise-skills-react/). Covers public, authenticated user, and authenticated admin flows, plus WCAG 2.1 AA accessibility audits and a read-only PROD smoke run. All credentials live in the single central project secret wise-application-skills; no long-lived credentials in GitHub Actions or in this repository.

Selector strategy — data-testid over translated labels

Selectors that depend on getByRole(..., { name: /<text>/i }) against a translated label silently break the moment the runner's locale changes. The login button is the canonical example: the i18n value of layout.login is Connexion (fr), Log in (en, with a space), Iniciar sesión (es), Anmelden (de), ログイン (ja), 登录 (zh). The previous regex /connexion|login/i only matched the French label because "Log in" does not contain the substring login — and the CI Chromium ships with accept-language: en-US by default, so all four E2E tests that touched the login flow timed out and cascaded into 50 skipped tests.

Rule of thumb for E2E selectors against this app:

  1. Prefer data-testid for any control whose label is user-visible and translated. Add data-testid="<scope>-<action>" on the React component (e.g. header-login-button, header-logout-button) and select it with page.getByTestId('header-login-button'). Locale-independent, stable across i18n updates, zero impact on UX or accessibility.
  2. Use getByRole for controls whose accessible role/name is stable across locales — typically generic shells like main, nav, dialog. The role itself does not depend on translation.
  3. Avoid regexes on translated labels. If you really must, list every supported locale and pin the runner's Accept-Language via use.locale in playwright.config.ts so the test surface stays deterministic.

The current data-testid taxonomy covers:

data-testid Component Used by
header-login-button AppShell.tsx (header) tests/auth/{user,admin}.setup.ts, tests/e2e/public/auth.spec.ts
header-logout-button AppShell.tsx (header) tests/e2e/user/notifications.spec.ts (@smoke : attente hydratation OIDC)
header-notifications-link AppShell.tsx (header) tests/e2e/user/notifications.spec.ts (@smoke : cloche notifications une fois authentifié)
skill-build-stepper SkillBuildPage.tsx tests/e2e/user/skill-build.spec.ts

Add a row to that table whenever you introduce a new test-id.

Retrieval at runtime

In CI, the workflow .github/workflows/e2e.yaml retrieves only the keys it needs for the target environment. The retrieval step never prints values; it masks them via ::add-mask:: and writes them to $GITHUB_ENV. Temporary files are shred -u-ed before the step exits.

runs-on: [atmos-aws-arc-runner-set]
container:
  image: public-mirror-docker-dockerhub.artifactory./node:20-bookworm
steps:
  - uses: -InnerSource/github-actions-library/setup-root-ca@main
  - uses: -InnerSource/github-actions-library/install-aws-cli@main
  - name: Configure AWS credentials (OIDC)
    uses: aws-actions/configure-aws-credentials@v4
    with:
      role-to-assume: arn:aws:iam::488440267176:role/App_InfraRunner_Github_Role
      aws-region: eu-west-1
  - name: Load E2E_*_INT credentials and BASE_URL_INT from wise-application-skills
    run: |
      aws secretsmanager get-secret-value --secret-id wise-application-skills ...
      # Exports the four E2E_*_INT keys (masked) and the BASE_URL_INT value
      # under the env name E2E_BASE_URL into $GITHUB_ENV.

Essensys internal PKI — full chain install in system trust + Chromium NSS

The three target hostnames (skills.{int,uat,prod}.) are signed by the Essensys internal root CA. The CI container ships an initial CA bundle (via -InnerSource/github-actions-library/setup-root-ca), exported through:

NODE_EXTRA_CA_CERTS=/usr/local/share/ca-certificates/-ca.crt
REQUESTS_CA_BUNDLE=/etc/ssl/certs/ca-certificates.crt
SSL_CERT_FILE=/etc/ssl/certs/ca-certificates.crt
CURL_CA_BUNDLE=/etc/ssl/certs/ca-certificates.crt

That covers Node, Python requests, OpenSSL CLIs, curl. It does not cover Chromium, which uses its own NSS database (sql:$HOME/.pki/nssdb) and ignores those env vars entirely. As a result, an unmodified container fails every Playwright navigation with net::ERR_CERT_AUTHORITY_INVALID.

The workflow therefore installs the full Essensys CA chain in both the system trust store and the Chromium NSS store. The chain itself is fetched at job time from AWS Secrets Manager (arn:aws:secretsmanager:eu-west-1:488440267176:secret:_CA_CHAIN_PEM_20230918-yr1xnv) so the canonical chain — managed centrally by the Essensys PKI team — is always current without checking certificates into Git.

The install is wrapped in a local composite action, .github/actions/install--ca-chain. Each E2E job invokes it once, right after Load E2E_*_<ENV> (which already assumed the IAM role):

- name: Install Essensys CA chain (system trust + Chromium NSS)
  uses: ./.github/actions/install--ca-chain

The action performs six steps:

  1. aws secretsmanager get-secret-value --secret-id _CA_CHAIN_PEM_20230918 → writes the PEM to /tmp/-ca-chain.pem (mode 644). The chain is not a secret per se (public certificates) but living in Secrets Manager gives a single rotation point owned by the Essensys PKI team.
  2. Copies the PEM to /usr/local/share/ca-certificates/-ca-chain.crt and runs update-ca-certificates, propagating the chain to /etc/ssl/certs/ca-certificates.crt for every system tool.
  3. apt-get install -y --no-install-recommends libnss3-tools to get certutil (Mozilla's NSS CLI).
  4. mkdir -p $HOME/.pki/nssdb && certutil -d sql:$HOME/.pki/nssdb -N --empty-password initialises the per-user NSS database (idempotent if the DB already exists, hence the || true).
  5. certutil -d sql:$HOME/.pki/nssdb -A -t "C,," -n -ca-chain -i /tmp/-ca-chain.pem imports the chain. Because the job runs as root (default in node:20-bookworm), HOME=/root and Chromium will read sql:/root/.pki/nssdb at launch time.
  6. certutil -L is then grepped for the -ca-chain entry; the step fails fast (::error::-ca-chain not present in NSS store after import) if the import was silently skipped. The PEM file is then rm -f-ed.

With this in place, ignoreHTTPSErrors is NOT enabled in playwright.config.ts. Chromium does a real chain-of-trust validation against the Essensys root CA, exactly like a real user.

IAM — the consumer role (App_InfraRunner_Github_Role) must hold secretsmanager:GetSecretValue on arn:aws:secretsmanager:eu-west-1:488440267176:secret:_CA_CHAIN_PEM_20230918-* in addition to the wise-application-skills-* ARN. If the workflow fails on Install Essensys CA chain with an AccessDenied, ask a CE_AppAdmin operator to extend the role's inline policy.

Single-secret policy exception — the project rule of .cursor/rules/-w-tools-bastion.mdc § 9 mandates that project secrets live in wise-application-skills. _CA_CHAIN_PEM_20230918 is a platform secret shared across every Essensys project, owned by the Essensys PKI team — not a project secret. The exception is therefore legitimate: the project secret is and remains wise-application-skills; the platform CA chain is consumed read-only.

PROD read-only safeguards

The PROD smoke job runs against the live customer environment, so we apply several layered guards:

  1. GitHub Environment gate. The playwright-prod-smoke job declares environment: production-smoke, which is configured with required reviewers and (recommended) a main-only branch protection. The job cannot start — and therefore cannot assume any AWS role — until a reviewer approves the run from the GitHub UI.
  2. Read-only Keycloak accounts. _prod_e2e_readonly user and admin accounts have no write permissions — even if the test code submits a mutation, the API rejects it.
  3. HTTP route guard. The custom test fixture in wise-skills-react/tests/e2e/_guards.ts registers page.route('**/api/**', ...) and throws on any non-GET/HEAD/ OPTIONS request when both E2E_ENV=PROD and E2E_READ_ONLY=true. Any attempted mutation surfaces immediately as a test failure.
  4. @smoke selector. The PROD job only runs tests tagged { tag: '@smoke' }. We have audited every tagged test to confirm it does not call any mutating API.
  5. a11y excluded. The a11y project is not part of the PROD selector to avoid burst traffic on the production stack.
  6. 30-day artifact retention on the Playwright report so any incident can be investigated after the fact.

Incident runbook

When playwright-prod-smoke fails, follow this sequence:

1. Confirm failure scope

  • Open the workflow run in GitHub Actions.
  • Download the playwright-report-prod-smoke artifact.
  • Inspect playwright-report/index.html for the failing test name, screenshot, trace and junit.xml.

2. Reproduce on UAT first

# Trigger the UAT job from the GitHub UI:
gh workflow run e2e.yaml -f target_env=UAT

UAT shares the same code path; if it reproduces there, the bug is in the application or the API and not in PROD configuration drift.

3. Investigate PROD via the bastion

All AWS / kubectl access goes through the bastion (passwordless SSH).

ssh  '
  set -e
  aws eks update-kubeconfig --region eu-west-1 --name w-c-eks-f-prod
  kubectl --context w-c-eks-f-prod get pods -n skills-portal -o wide
  kubectl --context w-c-eks-f-prod logs -n skills-portal \
    -l app.kubernetes.io/name=skills-portal --tail=200
'

For OIDC issues, check Keycloak directly via the W-Tools console.

4. Decide the next action

Symptom Action
API 5xx on /api/skills Page on-call, see docs/operations.md rollback procedure
Login redirect loop Verify Keycloak realm + redirect URIs, no rollback yet
Read-only account no longer has access Open a Keycloak ticket; mute the affected @smoke test for the run, retry
Selector drifted (UI change in PROD) Open a fix PR — INT/UAT must catch this before the next deploy

5. Communicate

  • Open an issue in essensys-hub/skills-portal with label prod-incident, attach the Playwright report URL.
  • Once root cause is known, update this runbook with the new symptom row.

Report archival on S3 + admin portal

After every Playwright run, the workflow archives the full report tree (HTML SPA, data/ videos/traces/screenshots, trace/ viewer, junit.xml) plus a fresh meta.json to the per-environment S3 bucket ewise-skills-<env> under the prefix skills-portal/<env>/e2e-reports/<github.run_id>/. The composite action .github/actions/archive-e2e-report-s3/action.yml runs after Upload Playwright report in each of the three jobs.

S3 layout

s3://ewise-skills-<env>/skills-portal/<env>/e2e-reports/<run_id>/
├── index.html                 # Playwright SPA entry point
├── junit.xml                  # parsed by API + admin UI
├── meta.json                  # schema_version=1, see below
├── data/
│   └── <hash>.{webm,zip,png}  # videos / traces / screenshots
└── trace/                     # Trace Viewer SPA (when produced)

meta.json (schema_version=1) carries run_id, env, branch, sha, triggered_by, started_at, ended_at, duration_seconds, conclusion (success/failure/cancelled), the JUnit summary (tests/failures/skipped/errors), and the GitHub Actions HTML URL.

Source of credentials

Both the CI uploader and the API pod use the same static IAM-user keys already used by the Backups feature, fetched from the central secret:

  • wise-application-skills.ACCESS_KEY_S3_<INT|UAT|PROD>
  • wise-application-skills.SECRET_KEY_S3_<INT|UAT|PROD>

The composite action assumes the existing App_InfraRunner_Github_Role via OIDC only to read the secret, then drops AWS_SESSION_TOKEN for the S3 calls so awscli falls back to the static-user identity. The override is scoped to a single shell — never written to $GITHUB_ENV — so subsequent steps still see the OIDC session.

Operator action — extending the IAM-user inline policy (one ticket per env)

The IAM user behind ACCESS_KEY_S3_<ENV> already holds s3:{Put,Get,Delete,List,Head}* on arn:aws:s3:::ewise-skills-<env>/skills-portal/<env>/backups/*. We need to extend that to the new e2e-reports/* prefix. No change to App_InfraRunner_Github_Role is required — the OIDC role only reads the secret.

       "Resource": [
         "arn:aws:s3:::ewise-skills-<env>",
-        "arn:aws:s3:::ewise-skills-<env>/skills-portal/<env>/backups/*"
+        "arn:aws:s3:::ewise-skills-<env>/skills-portal/<env>/backups/*",

Until the operator confirms each environment, the workflow step is marked continue-on-error: true so a missing policy never regresses the workflow's green status. Once confirmed, remove the flag — see the inline # TODO comments in .github/workflows/e2e.yaml.

Admin portal

The new admin pages are mounted at:

  • /admin/e2e-reports — table of archived runs across INT/UAT/PROD, with env filter, summary chips, and a delete action.
  • /admin/e2e-reports/<run_id> — run header, Summary tab (junit summary cards + per-testcase table) and Full report tab (Playwright SPA embedded in an iframe served by GET /api/e2e-reports/<run_id>/file/index.html).

When the IAM-user inline policy has not been extended yet, the API returns 200 + {runs: [], warning: "s3_access_not_configured"} for the list endpoint and 503 for the file endpoints. The admin UI surfaces a friendly banner instead of an error toast.

Manual cleanup

There is no automatic retention. To delete a single run, use the admin UI Delete button (typed run-id confirmation) or:

aws s3 rm \
  "s3://ewise-skills-<env>/skills-portal/<env>/e2e-reports/<run_id>/" \
  --recursive

To bulk-delete every run older than 30 days, list, filter, and pipe to aws s3 rm:

aws s3 ls "s3://ewise-skills-<env>/skills-portal/<env>/e2e-reports/" \
  | awk -v d="$(date -u -d '30 days ago' +%Y-%m-%d)" '$1 < d {print $2}' \
  | while read -r p; do
      aws s3 rm \
        "s3://ewise-skills-<env>/skills-portal/<env>/e2e-reports/${p}" \
        --recursive
    done
